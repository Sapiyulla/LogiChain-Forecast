**Концепт программного средства «LogiChain Forecast»**

---

### 1. Назначение системы

Программное средство предназначено для прогнозирования загрузки складских площадей (количество занятых паллето-мест) на горизонте до 4 недель. Целевой пользователь — менеджер по планированию складских ресурсов 3PL-оператора. Система автоматизирует полный цикл работы с данными: генерацию синтетической истории, ETL-обработку, построение прогнозной модели и визуализацию результатов.

---

### 2. Бизнес-требования

| Идентификатор | Требование |
| :--- | :--- |
| BR-01 | Система должна предоставлять прогноз загрузки склада на каждые сутки в пределах горизонта прогнозирования. |
| BR-02 | Прогноз должен обновляться еженедельно по пятницам с учётом новых данных за прошедшую неделю. |
| BR-03 | Система должна визуализировать фактические исторические данные и прогнозные значения на едином графике. |
| BR-04 | Система должна формировать PDF-отчёт с ключевыми показателями прогноза для передачи руководству. |
| BR-05 | Система должна предоставлять оценку качества прогноза (MAE, MAPE) на ретроспективных данных. |

---

### 3. Функциональные требования

#### 3.1. Генерация синтетических данных
- **FR-01:** Скрипт генерации создаёт CSV-файл с почасовыми записями занятости склада за 3 года.
- **FR-02:** Генерируемый ряд должен содержать:
  - Недельную и месячную сезонность.
  - Тренд (имитация роста бизнеса клиента).
  - Всплески в периоды праздников и маркетинговых акций.
  - Случайный шум и единичные пропуски данных (эмуляция сбоев датчиков).

#### 3.2. Оркестрация (Apache Airflow)
- **FR-03:** DAG Airflow должен выполняться по расписанию `0 18 * * 5` (пятница, 18:00).
- **FR-04:** DAG выполняет последовательность задач:
  1. Генерация 1000–2000 новых записей за прошедшую неделю.
  2. Загрузка новых данных в слой Raw хранилища.
  3. Запуск трансформаций dbt.
  4. Переобучение модели прогнозирования на обновлённой витрине.
  5. Генерация PDF-отчёта.

#### 3.3. Трансформация данных (dbt)
- **FR-05:** Слой Raw содержит таблицу `raw_warehouse_occupancy` без изменений.
- **FR-06:** Слой Staging включает модель `stg_warehouse_occupancy` с:
  - Приведением типов.
  - Заполнением пропусков методом линейной интерполяции.
  - Фильтрацией явных выбросов (значения за границами 3σ).
- **FR-07:** Слой Analytics включает витрину `fct_forecast_features` с расчётными признаками:
  - Лаги (1, 7, 30 дней).
  - Скользящие средние (7 и 30 дней).
  - Признак дня недели (one-hot).
  - Признак близости праздника (бинарный, ±3 дня).
- **FR-08:** dbt генерирует документацию (`dbt docs generate`) с описанием всех полей и lineage.

#### 3.4. Модель прогнозирования
- **FR-09:** Используется модель градиентного бустинга LightGBM.
- **FR-10:** Целевая переменная — `occupied_pallets`.
- **FR-11:** Горизонт прогнозирования — 28 дней.
- **FR-12:** Модель сохраняется в формате `.pkl` после каждого переобучения.

#### 3.5. Пользовательский интерфейс (Streamlit)
- **FR-13:** Дашборд отображает:
  - Интерактивный график Plotly с историческими данными и прогнозом.
  - Таблицу с прогнозными значениями на ближайшие 7 дней.
  - Метрики качества модели (MAE, MAPE) на тестовой выборке.
- **FR-14:** Кнопка «Сгенерировать отчёт» инициирует создание PDF и его скачивание.

#### 3.6. Формирование отчётности
- **FR-15:** PDF-отчёт содержит:
  - Дату и время формирования.
  - График прогноза.
  - Таблицу прогноза на следующую неделю.
  - Сводку по ожидаемой пиковой загрузке.

### 4. Нефункциональные требования

| Идентификатор | Требование |
| :--- | :--- |
| NFR-01 | Все компоненты системы упакованы в Docker-контейнеры и запускаются единой командой `docker compose up`. |
| NFR-02 | Хранилище данных — PostgreSQL 15. |
| NFR-03 | Код на Python должен соответствовать PEP8, проверка линтером Ruff. |
| NFR-04 | Время выполнения полного цикла DAG (генерация → прогноз) не должно превышать 5 минут на тестовом объёме данных (≈50 000 записей). |
| NFR-05 | Документация по развёртыванию и использованию представлена в README.md репозитория. |

### 5. Архитектура системы

#### 5.1. Диаграмма потоков данных (DFD, уровень контекста)

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  Генератор  │────▶│   PostgreSQL     │────▶│   dbt (модели)  │
│ синтетики   │     │   (Raw → Stage   │     │   Трансформации │
└─────────────┘     │    → Analytics)  │     └────────┬────────┘
                    └──────────────────┘              │
                            ▲                         ▼
                            │               ┌─────────────────┐
┌─────────────┐      ┌──────┴───────┐       │   ML-скрипт     │
│   Airflow   │─────▶│   Загрузка   │       │  (LightGBM)     │
│   DAG       │      │   новых      │       └────────┬────────┘
└─────────────┘      │   данных     │                │
                     └──────────────┘                ▼
                                            ┌─────────────────┐
                                            │   Streamlit     │
                                            │   Dashboard     │
                                            └────────┬────────┘
                                                     │
                                                     ▼
                                            ┌─────────────────┐
                                            │   PDF-отчёт     │
                                            └─────────────────┘
```

#### 5.2. Структура Docker-сервисов

| Сервис | Порт | Назначение |
| :--- | :--- | :--- |
| `postgres` | 5432 | Хранилище данных |
| `airflow-webserver` | 8080 | Интерфейс управления DAG |
| `airflow-scheduler` | — | Исполнение задач по расписанию |
| `streamlit` | 8501 | Пользовательский дашборд |
| `dbt` | — | Выполняется как задача Airflow (не сервис) |

| Визуализация | Streamlit | Быстрая разработка интерактивных дашбордов |

### 7. План-график реализации

| Неделя | Этап | Ожидаемые артефакты |
| :--- | :--- | :--- |
| 1 | Проектирование | DFD-диаграмма, схема БД, `docker-compose.yml` |
| 2 | Генератор + ETL | Скрипт генерации, DAG загрузки, dbt-модели слоя Staging |
| 3 | ML + Визуализация | Модель LightGBM, Streamlit-дашборд, PDF-генератор |
| 4 | Интеграция и отладка | Полный цикл DAG, документация, подготовка демо-сценария |

---

### 8. Критерии приёмки

| Критерий | Способ проверки |
| :--- | :--- |
| Запуск одной командой | `docker compose up`, все сервисы в статусе healthy |
| Выполнение DAG | Успешный запуск в Airflow, появление новых записей в БД |
| Обновление прогноза | Изменение графика в Streamlit после DAG |
| Качество модели | MAPE < 15% на тестовой выборке (контролируется в логах) |
| Генерация PDF | Файл формируется и содержит корректный график и таблицу |
| Документация dbt | Доступна по ссылке `target/index.html` после `dbt docs generate` |
